## 3. Routing Approach

### URL Structure

The site uses **locale-prefixed routes** for all non-default languages. English (the default) can be accessed with or without the `/en` prefix.

| Language | URL Pattern | Example |
|----------|-------------|---------|
| English | `/` or `/en/*` | `/contact-us` or `/en/contact-us` |
| Spanish | `/es/*` | `/es/contact-us` |
| Armenian | `/hy/*` | `/hy/contact-us` |
| Russian | `/ru/*` | `/ru/contact-us` |
| Ukrainian | `/uk/*` | `/uk/contact-us` |

### Implementation Strategy

Since this project uses React with client-side routing (Wouter), the i18n implementation uses a **client-side locale context** approach rather than Next.js App Router's built-in i18n (which requires server components).

```
/                     → English (default)
/es                   → Spanish home
/hy                   → Armenian home
/ru                   → Russian home
/uk                   → Ukrainian home
/es/practice-areas    → Spanish practice areas
/hy/contact-us        → Armenian contact page
```

### Route Configuration

```typescript
// Locale configuration
export const locales = ['en', 'es', 'hy', 'ru', 'uk'] as const;
export type Locale = typeof locales[number];
export const defaultLocale: Locale = 'en';

// Route helper
export function localizedPath(path: string, locale: Locale): string {
  if (locale === defaultLocale) return path;
  return `/${locale}${path}`;
}
```

## 4. Persistence Method

User language preference is persisted using **localStorage** with a cookie fallback for SSR compatibility.

### Storage Key

```typescript
const LOCALE_STORAGE_KEY = 'gurovich-locale';
```

### Persistence Logic

1. **On language selection:** Save to localStorage and set a cookie
2. **On page load:** Check localStorage first, then cookie, then browser `navigator.language`
3. **On refresh:** Restore from persisted preference
4. **Cookie expiry:** 1 year

### Implementation

```typescript
// Save preference
function setLocalePreference(locale: Locale) {
  localStorage.setItem(LOCALE_STORAGE_KEY, locale);
  document.cookie = `${LOCALE_STORAGE_KEY}=${locale};path=/;max-age=31536000`;
}

// Get preference
function getLocalePreference(): Locale {
  // Check localStorage
  const stored = localStorage.getItem(LOCALE_STORAGE_KEY);
  if (stored && locales.includes(stored as Locale)) {
    return stored as Locale;
  }
  
  // Check cookie
  const cookie = document.cookie
    .split('; ')
    .find(row => row.startsWith(LOCALE_STORAGE_KEY));
  if (cookie) {
    const value = cookie.split('=')[1];
    if (locales.includes(value as Locale)) {
      return value as Locale;
    }
  }
  
  // Check browser language
  const browserLang = navigator.language.split('-')[0];
  if (locales.includes(browserLang as Locale)) {
    return browserLang as Locale;
  }
  
  return defaultLocale;
}
```

## 5. Translation File Structure

### Directory Layout

```
client/src/
└── i18n/
    ├── index.ts           → i18n configuration and hooks
    ├── locales/
    │   ├── en.json        → English translations
    │   ├── es.json        → Spanish translations
    │   ├── hy.json        → Armenian translations
    │   ├── ru.json        → Russian translations
    │   └── uk.json        → Ukrainian translations
    └── types.ts           → TypeScript definitions
```

### Translation File Format

Each locale file is a flat JSON object with dot-notation keys for namespacing:

```json
{
  "nav.home": "Home",
  "nav.practiceAreas": "Practice Areas",
  "nav.ourTeam": "Our Team",
  "nav.testimonials": "Testimonials",
  "nav.blog": "Blog",
  "nav.contactUs": "Contact Us",
  
  "header.phone": "(818) 401-4725",
  "header.email": "kg@gurovichlaw.com",
  "header.hours": "Mon-Fri 9:00 AM - 5:00 PM",
  
  "hero.headline": "VIGOROUS ADVOCACY",
  "hero.subheadline": "FOR LIFE'S MOST SERIOUS LEGAL CHALLENGES",
  "hero.cta.call": "CALL TODAY",
  "hero.cta.consultation": "Free Consultation",
  
  "footer.copyright": "© 2026 Gurovich Law Group, APC. All rights reserved.",
  "footer.privacyPolicy": "Privacy Policy",
  "footer.termsOfService": "Terms of Service",
  "footer.disclaimer": "Disclaimer"
}
```

### Key Naming Conventions

| Prefix | Usage | Example |
|--------|-------|---------|
| `nav.*` | Navigation labels | `nav.practiceAreas` |
| `header.*` | Header elements | `header.phone` |
| `footer.*` | Footer elements | `footer.copyright` |
| `hero.*` | Hero section | `hero.headline` |
| `cta.*` | Call-to-action buttons | `cta.getStarted` |
| `form.*` | Form labels and messages | `form.name.label` |
| `page.*` | Page-specific content | `page.about.title` |
| `practice.*` | Practice area content | `practice.personalInjury.title` |

## 6. How to Add/Maintain Translations

### Adding a New Translation Key

1. **Add the key to `en.json` first** (English is the source of truth)
2. **Add translations to all other locale files** with the same key
3. **Run the missing keys check** to verify completeness

```bash
# Check for missing translation keys
pnpm run i18n:check
```

### Adding a New Language

1. Create a new locale file in `client/src/i18n/locales/` (e.g., `fr.json`)
2. Add the locale code to the `locales` array in `client/src/i18n/index.ts`
3. Add the language to the `LanguageSelector` component
4. Copy all keys from `en.json` and translate

### Translation Workflow

```
1. Developer adds new UI text
   ↓
2. Add key to en.json with English text
   ↓
3. Add placeholder to other locales (copy English)
   ↓
4. Mark for translation review
   ↓
5. Professional translator updates locale files
   ↓
6. PR review and merge
```

### Missing Key Fallback

If a translation key is missing in a non-English locale, the system automatically falls back to English and logs a warning:

```typescript
function t(key: string): string {
  const translation = translations[currentLocale][key];
  if (!translation) {
    console.warn(`Missing translation: ${key} for locale: ${currentLocale}`);
    return translations['en'][key] || key;
  }
  return translation;
}
```

## 7. Blog Translation Strategy

### Approach: Language-Specific MDX Files

Blog posts support full translations through separate MDX files per language:

```
content/blog/
├── hidden-penalties-criminal-conviction/
│   ├── en.mdx    → English version
│   ├── es.mdx    → Spanish version (if available)
│   └── meta.json → Shared metadata
└── tips-insurance-adjusters/
    ├── en.mdx
    └── meta.json
```

### Fallback Behavior

When a blog post is not available in the user's selected language:

1. **Display the English version** as fallback
2. **Show a prominent notice** at the top of the post:

```
┌─────────────────────────────────────────────────────────┐
│ ℹ️ This article is currently available in English only. │
│    Translation coming soon.                             │
└─────────────────────────────────────────────────────────┘
```

### Legal Content Warning

Legal blog content is **never auto-translated**. All translations must be:

1. Professionally translated by a qualified translator
2. Reviewed by legal staff for accuracy
3. Clearly labeled with the translation date

```
┌─────────────────────────────────────────────────────────┐
│ ⚠️ LEGAL NOTICE: This translation is provided for       │
│    informational purposes only. For legal advice,       │
│    please consult with our attorneys directly.          │
│    Translation date: January 28, 2026                   │
└─────────────────────────────────────────────────────────┘
```


The `<html>` element's `lang` attribute is dynamically set based on the current locale:

```html
<html lang="es">
```

The sitemap includes all language versions of each page with proper `hreflang` annotations.
